# Python 代码注释（中文统一规范）

## 核心目标

让代码在 **半年后回看 / 交接 / 排查问题** 时依然可读：注释解释 **意图、假设、边界、风险**，而不是复述代码字面。

## 适用范围与硬性要求（全覆盖）

- **模块级注释（每个 `.py` 文件必须有）**
  - 说明：这个文件做什么、输入/输出是什么、如何运行、关键注意事项
- **`import` 代码块后说明（每个 `.py` 文件必须有）**
  - 在 `import` 语句块结束后，必须空一行并追加“文件实现说明”注释
  - 说明必须包含：当前文件核心职责；若涉及流程，按顺序写清流程步骤（如 1 -> 2 -> 3）
- **类/函数 docstring（每个 `class`/`def` 必须有）**
  - **至少包含**：`参数`、`返回`
  - **按需包含**：`异常`、`示例`、`注意`
- **关键逻辑行内注释（必须有）**
  - 对“重要但不显眼”的决定写清楚 **为什么**

> 语言要求：**只用中文**（变量/参数名保留英文即可）。

## 写作规则（优先级从高到低）

1. **优先写“为什么/约束/副作用”**，少写“这行在干嘛”
2. **阈值/魔法数必须说明含义**（来源、经验值、口径）
3. **数据口径必须显式**（日期、时区、复权、频率、交易日、缺失值处理）
4. **边界条件必须说明**（空数据、重复数据、异常格式、接口失败、限流/配额）
5. **风险点必须提示**（数据泄漏/未来函数、幸存者偏差、lookahead、复权口径不一致）
6. **保持一致性**：同一概念用同一套中文术语（例如：配额/流量/增量更新/落盘/缓存）

## 统一模板（直接复制）

完整模板与示例请见：`references/templates.md`。

最低要求：

- 模块：一句话 + 关键点列表 + 运行方式（或入口函数）
- 函数/方法：一句话 + 参数 + 返回（异常/示例按需）
- 类：职责边界 + 关键属性/方法说明（示例按需）

## 行内注释“必写点位清单”

- **关键分支**：为什么走这个分支、另一分支的代价是什么
- **去重/合并**：用什么 key、为什么 `keep='last'`、是否会覆盖历史
- **数据清洗**：为什么 dropna、为什么时间对齐、为什么截断到日粒度
- **外部依赖**：网络/数据库/接口失败时怎么降级、怎么重试
- **性能与资源**：为什么批量/缓存/分批/限速，避免什么问题

## 输出要求（执行本技能时的约束）

- **不改变业务逻辑** 的前提下补注释
- 注释保持“短而准”：一个逻辑块 1-3 行解释即可
- 对外暴露的 API（别人会调用的函数）docstring 更完整；内部函数可略少但仍必须有

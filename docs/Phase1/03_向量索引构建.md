# T3 — 向量索引构建

> 使用 UltraRAG 的 `retriever` Server，以 Google `text-embedding-004` 对 `chunks.jsonl` 向量化，再用 FAISS 建立本地索引。

---

## 流程图（调整后）

`text-embedding-004` 已停用，新模型（`gemini-embedding-001`）使用原生 API，不经过 UltraRAG 的 `retriever_embed` 步骤。流程拆为两段：

```
chunks.jsonl
(含 images 字段)
     │
     ▼
[Step A] google_embedder.py         ← 自定义：调 Google 原生 API
     │    批量 embed → L2 归一化
     ▼
embedding.npy
data/kb/agv_demo/embedding/
     │
     ▼
[Step B] retriever.retriever_index  ← UltraRAG FAISS 建索引
     │    读 embedding.npy → 建 IndexFlatIP
     ▼
index.index
data/kb/agv_demo/index/
```

## Pipeline YAML 文件

### `examples/agv_index_only.yaml`（只跑 index，不跑 embed）

```yaml
# AGV 知识库 — FAISS 索引构建 Pipeline（embed 已由 google_embedder.py 完成）

servers:
  retriever: servers/retriever

pipeline:
- retriever.retriever_init
- retriever.retriever_index
```

### 完整建索引命令（两步）

```bash
# Step A: Google API 批量 embedding
python -c "
from custom_app.services.google_embedder import build_embedding_npy
build_embedding_npy(
    chunks_jsonl='data/kb/agv_demo/corpora/chunks.jsonl',
    output_npy='data/kb/agv_demo/embedding/embedding.npy'
)
"

# Step B: UltraRAG FAISS 建索引
ultrarag run examples/agv_index_only.yaml
```

## Google Embedding API 限制与应对

| 限制项 | `gemini-embedding-001` | 应对措施 |
|--------|----------------------|---------|
| 每分钟请求数（RPM） | 免费层 1500 RPM | `BATCH_SIZE=20`，批次间 `RATE_DELAY=0.1s` |
| batchEmbedContents 每批上限 | 100 条/请求 | 保守取 20 条，速率安全 |
| 单条文本最大 token | 2048 | chunk_size=500 字符远低于上限，安全 |
| 输出向量维度 | 默认 3072，可截断 | 配置 `outputDimensionality=768`，节省 4x 存储 |
| 向量是否归一化 | 3072 维已归一，其他维需手动归一 | `google_embedder.py` 已做 L2 归一化 |

遇到 429 速率限制时，`google_embedder.py` 已内置指数退避重试（最多 3 次）。若仍频繁限流，调大 `RATE_DELAY`：

```python
# custom_app/services/google_embedder.py
RATE_DELAY = 0.5   # 从 0.1 调到 0.5
BATCH_SIZE = 10    # 从 20 调到 10
```

## FAISS 索引说明

UltraRAG 默认使用 `IndexFlatIP`（内积，即余弦相似度变体），适合 `normalize_embeddings=true` 的向量。

```
data/kb/agv_demo/
├── embedding/
│   └── embedding.npy        ← 所有 chunk 的向量矩阵 (N × 768)
└── index/
    └── index.index          ← FAISS 索引文件（含向量 + 位置映射）
```

FAISS 索引是**纯本地文件**，无需额外进程，进程启动时加载到内存（约 100 份文档 × 平均 100 chunks × 768 × 4 bytes ≈ 30 MB，完全可以接受）。

## 验证索引质量

建索引完成后，用以下脚本验证召回效果：

```python
# scripts/test_retrieval.py
"""验证 FAISS 向量检索效果"""
import asyncio
import os
from dotenv import load_dotenv

load_dotenv()

async def test():
    # 直接调用 UltraRAG API
    from ultrarag.api import run_pipeline

    results = await run_pipeline(
        pipeline_yaml="examples/agv_rag.yaml",
        inputs={
            "q_ls": ["换电操作的注意事项有哪些"],
            "top_k": 5,
        }
    )
    for r in results.get("ret_psg", [[])[0]:
        print(f"[{r.get('score', 0):.4f}] {r.get('title', '')} — {r.get('contents', '')[:80]}")
        print(f"  images: {r.get('images', [])}")

asyncio.run(test())
```

**预期结果**：
- score > 0.7 的 chunk 与查询语义相关  
- 含图片段落的 chunk `images` 字段不为空  
- 无关 chunk 排在后面（score < 0.5）

## 增量更新策略（Phase 1 说明）

Phase 1 每次新增文档后**全量重建**索引（流程简单、文档量小时性能可接受）：

```bash
# 1. 重新解析所有文档
python custom_app/services/docx_parser.py --input data/kb/agv_demo/raw/ ...

# 2. 重建向量索引（覆盖旧文件）
ultrarag run examples/agv_index.yaml
```

增量更新（只对新文档做 embed + 追加索引）是 Phase 2/3 的优化项。

## 验收标准（T3）

1. `data/kb/agv_demo/index/index.index` 文件存在，大小非零。  
2. `data/kb/agv_demo/embedding/embedding.npy` 行数与 `chunks.jsonl` 行数**完全一致**。  
3. 检索测试脚本输出前 3 条相关结果，`score > 0.6`。  
4. 建索引全程无 HTTP 4xx/5xx 报错（偶发 429 重试成功也算通过）。

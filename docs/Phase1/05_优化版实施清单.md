# Phase 1.5 — 回答质量优化实施清单

> 目标：在不破坏当前可用链路的前提下，解决“回答啰嗦、图文不匹配”两类核心问题，并形成可复现的对比验证流程。

---

## 0. 范围与原则

- 范围：仅优化 `custom_app` 问答链路，不重构 Phase 2 API。
- 原则：小步快跑、每步可回滚、每步可验证。
- 成功标准：质量提升可量化，不以主观“感觉更好”作为唯一依据。

---

## 1. 实施路线（按优先级）

### P0（必须）

1. 接入 `rewrite_query`（单次 LLM 改写）
2. 调整召回策略：`recall_top_k` 提高；**步骤类 / SOP** 场景已演进为「主文档全量扩展 + `final_top_k: 0` 不截断」，与早期「压到 3 条再生成」不同（以步骤完整为先）。
3. 保持现有 `sources` 审计字段不变
4. 增加对比测试样例（优化前 vs 优化后）

### P1（强烈建议）

5. 接入本地 reranker（GPU 优先，CPU 可退化）
6. Prompt 增加强约束：关键结论必须附 `source_id`
7. `answer_blocks` 图文绑定由关键词匹配升级为 `source_id` 显式绑定

### P2（可选增强）

8. 去重与近重复压缩（减少重复 chunk）
9. 低召回场景 query expansion（仅在召回不足时触发）
10. 轻量 MMR（提升多样性，减少“同一文档反复引用”）

---

## 2. 任务分解与验收标准

## T1 — Query Rewrite

- [x] 在 `rag_runner` 增加 `rewrite_query`（内部 `_rewrite_query`，响应字段 `rewrite_query`）
- [x] 设置失败兜底：改写失败时回退原始问题
- [x] 可观测：`POST /api/chat` 返回 JSON 中的 `rewrite_query`（原问题以请求体为准）

验收：
- [ ] 对同一问题，改写后的检索命中更聚焦（人工抽样 5 题）
- [ ] 改写失败不影响主链路返回

## T2 — Recall / 截断 / SOP 扩展（与实现对齐）

- [x] `servers/retriever/parameter.yaml`：`recall_top_k`（当前示例为 20）
- [x] `final_top_k: 0` 表示生成前**不**按条数截断命中列表（SOP 全步骤优先）
- [x] 步骤类问题：同主 `doc` 全量扩展 + 主文档消歧；`sources` 条数 = 该主文档下参与生成的 chunk 数（非固定 3）

验收（请按**当前策略**勾选，勿再要求「sources 必须 = 3」）：
- [ ] 换电等流程题：`meta.sop_full_doc_expand == true` 时步骤连续、无无关 doc 混入
- [ ] `meta.recall_top_k` / `meta.final_top_k` 与配置一致

## T3 — 本地 Rerank（GPU）

- [x] 配置项：`servers/retriever/parameter.yaml` → `rag_rerank`（`enabled`、`model_name_or_path`、`device`、`batch_size`、`trust_remote_code`）
- [x] GPU 不可用时：加载失败则尝试 CPU；仍失败则跳过重排并返回 `meta.rerank_skip_reason`
- [x] `meta`：`rerank_applied`、`rerank_ms`、`rerank_skip_reason`、`rerank_device`；日志见 `rag_rerank applied ...`

验收：
- [ ] `enabled: true` 且依赖齐全时：`meta.rerank_applied == true`，`meta.rerank_ms` 合理
- [ ] 关闭或缺依赖时：`meta.rerank_skip_reason` 可读，问答仍 200
- [ ] 多题抽样：主文档选择更稳（与 `_narrow_expand_docs` 协同）

## T4 — Prompt 引用约束

- [ ] Prompt 明确要求“仅依据给定 sources”
- [ ] 关键结论后附 `source_id`
- [ ] 无依据时输出明确兜底语句

验收：
- [ ] 人工检查 10 条回答，均可追溯到 `source_id`
- [ ] 幻觉内容明显下降

## T5 — 图文绑定升级（source_id）

- [ ] 每个文本 step 标注来源 `source_id`
- [ ] 图片仅绑定到同 `source_id` 的 step
- [ ] 无匹配图片时不强行展示

验收：
- [ ] 图文错配率下降（抽样 20 个 step）
- [ ] 不再出现“无关图片插入正确步骤”的问题

## 3. 对比测试计划（必须执行）

测试集建议（先 10 题）：
- 5 题“步骤类问题”（如换电流程、安全确认）
- 3 题“细节类问题”（参数、条件、异常处理）
- 2 题“无答案问题”（文档不存在）

每题采集以下指标：
- `answer_length`（字符数）
- `source_count`
- `citation_coverage`（结论可追溯比例）
- `image_alignment`（图文匹配是否正确）
- `latency_ms`

结论门槛（建议）：
- **SOP 全量扩展场景**：以「步骤完整、无串 doc、图文同源」为主，不强制 `source_count` / `answer_length` 下降（与早期「压到 3 条」策略不同）。
- 非 SOP 短答场景：仍可参考 `source_count` / `answer_length` 与延迟。
- `citation_coverage` >= 90%（T4 后强化）
- `image_alignment` >= 85%
- `latency_ms` 增幅可控（按本机可接受阈值）

## 附录 A — T1 / T2 验证步骤（`POST /api/chat`）

**不要用** `/api/chat/markdown` 验 T1/T2（无 `meta` / `rewrite_query` 字段）。

### A.1 T1：Query Rewrite

1. 启动 `custom_app`，对 `http://127.0.0.1:8080/api/chat` 发 JSON（示例）：

```json
{ "kb_id": "agv_demo", "question": "更换AGV电池的步骤?" }
```

2. 检查响应：
   - 存在 **`rewrite_query`**：应为便于检索的短查询（多为英文关键词组合）。
   - **原问题**：以请求体中的 `question` 为准（若需在 JSON 内一并返回，可后续加字段）。

3. **失败兜底**（可选）：暂时断开 vLLM 或填错 `servers/generation/parameter.yaml` 的 `base_url`，预期仍返回 200 时，`rewrite_query` 应退回与问题等价或原问句（不应因改写整请求失败）。

### A.2 T2：召回、截断、SOP 扩展

同一响应中检查：

| 字段 | 含义 |
|------|------|
| `meta.recall_top_k` | 与 `parameter.yaml` 中 `recall_top_k` 一致（或请求体覆盖 `top_k` 时取该值） |
| `meta.final_top_k` | 扩展后的 `hit` 条数（`final_top_k: 0` 时不做截断） |
| `meta.sop_full_doc_expand` | 是否触发了同 doc 全量扩展 |
| `meta.primary_expanded_doc` | 主文档名（多 doc 竞争时由首轮命中计数决定） |
| `sources` | 条数 = 最终参与生成的 chunk 数；每项含 `doc`、`source_id`、`excerpt` 等 |

**PowerShell 示例：**

```powershell
$body = @{ kb_id = "agv_demo"; question = "更换AGV电池的步骤?" } | ConvertTo-Json
$r = Invoke-RestMethod -Uri "http://127.0.0.1:8080/api/chat" -Method Post `
  -Body $body -ContentType "application/json; charset=utf-8" -TimeoutSec 300
$r.rewrite_query
$r.meta | ConvertTo-Json
$r.sources.Count
```

## 附录 B — T3：本地 Rerank 启用与验证

### B.1 依赖

```bash
pip install sentence-transformers torch
```

（有 NVIDIA GPU 时建议安装与显卡匹配的 **CUDA 版 torch**；仅 CPU 则 `device: cpu` 或 `auto` 在无 CUDA 时走 CPU。）

### B.2 配置

编辑 `servers/retriever/parameter.yaml`：

```yaml
rag_rerank:
  enabled: true
  model_name_or_path: openbmb/MiniCPM-Reranker-Light
  device: auto
  batch_size: 8
```

重启 Flask / `RagRunner.init()` 所在进程（切换 `kb_id` 会新建 `RagRunner` 也会重新读配置）。

### B.3 验证

再次 `POST /api/chat`，检查 `meta`：

- **`rerank_applied: true`**：重排已执行；**`rerank_device`** 为 `cuda` 或 `cpu`；**`rerank_ms`** 为耗时（毫秒）。
- **`rerank_applied: false`**：看 **`rerank_skip_reason`**：`disabled` / `sentence_transformers:...` / `predict_failed:...` 等。
- 首次加载模型会较慢，属正常；显存不足时会打日志并尝试 CPU。

### B.4 回滚

将 `rag_rerank.enabled` 设为 `false` 并重启即可关闭，主链路不依赖 rerank。

## 5. 风险与回滚

- 风险：rerank 模型过重导致时延过高  
  回滚：切轻量模型或临时关闭 rerank

- 风险：rewrite 过度改写导致偏题  
  回滚：加入“保守改写”模板，异常时直接回退原问题

- 风险：图文绑定改造影响前端渲染  
  回滚：保留旧 `sources.images` 渲染路径作为兼容分支

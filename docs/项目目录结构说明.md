# UltraRAG 项目目录结构说明（当前阶段）

> 目的：快速定位 Phase 1 AGV 私有知识库最小闭环相关代码与 API 入口。

## 1) 当前最小闭环主路径

```text
custom_app/
  app.py                         # Flask 启动入口（create_app + 注册 Blueprint）
  api/
    chat.py                      # 聊天接口路由（/api/chat, /api/chat/markdown, /health）
  services/
    docx_parser.py               # DOCX 解析 + 图片导出 + chunks.jsonl 生成
    google_embedder.py           # Google embedding（文档向量/查询向量）
    rag_runner.py                # 检索+生成编排（业务逻辑层，不是 API 路由层）

data/
  kb/agv_demo/
    corpora/chunks.jsonl
    embedding/embedding.npy
    index/index.index
    images/

prompt/
  agv_qa_rag.jinja               # AGV 问答模板（当前英文策略）

servers/
  generation/parameter.yaml      # vLLM 配置
  retriever/parameter.yaml       # 检索/索引配置

examples/
  agv_index_only.yaml            # 仅建索引 pipeline
  agv_rag.yaml                   # 问答 pipeline 参考

docs/
  AGV_RAG_系统方案.md
  Phase1/
    README.md
    02_DOCX解析入库.md
    03_向量索引构建.md
    04_问答流程.md
```

## 2) `custom_app` API 定位

- 你现在在用的两个接口：
  - `POST /api/chat`
  - `POST /api/chat/markdown`
  定义在 `custom_app/api/chat.py`。
- `custom_app/app.py` 负责应用创建与 Blueprint 注册（入口职责）。
- `custom_app/services/rag_runner.py` 是服务层，只负责：
  - query embedding
  - FAISS 检索
  - Prompt 组装
  - vLLM 生成
  - 结果结构化（`answer / answer_blocks / sources`）
  不负责 Flask 路由注册。

## 3) 仓库里“另一套 API”位置（UI 后端）

- 项目还包含 UI 后端：`ui/backend/app.py`，内部有较完整的 `/api/*` 路由集合（pipeline、kb、prompts 等）。
- 这套与 `custom_app/app.py` 的最小验证 API 是并行关系：
  - `custom_app/app.py`：Phase 1 最小闭环验证
  - `ui/backend/app.py`：平台化 UI 后端能力

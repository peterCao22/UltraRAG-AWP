# Phase 2 — 任务编排与状态流转

> 目标：将文档解析、Embedding、索引构建三段串成可追踪的任务流，并提供完整的状态查询与干预接口。

---

## 1. 任务生命周期

```
pending → running → success
                 ↘ failed → (retry) → pending → ...
pending → cancelled
```

| 状态 | 含义 |
|------|------|
| `pending` | 任务已创建，等待执行 |
| `running` | 正在执行（解析/embedding/建索引） |
| `success` | 全部步骤完成，索引可用 |
| `failed` | 执行中途失败，`last_error` 记录原因 |
| `cancelled` | 用户主动取消（已完成/运行中的任务不可逆） |

---

## 2. 三阶段串行流程

```
POST /api/kb/{kb_id}/ingest
        │
        ▼
  [1] 解析 DOCX → chunks.jsonl
        │
        ▼
  [2] chunks → embedding.npy（Google Embedding API）
        │
        ▼
  [3] embedding.npy → FAISS index.index
        │
        ▼
  更新 kb_jobs.status = 'success'
  更新 kb_documents.status = 'indexed'
  更新 knowledge_bases.last_indexed_at
```

每个阶段失败均跳到 `failed`，不做局部回滚（文件系统幂等：重试时覆盖写入）。

---

## 3. 同步 vs 异步执行模式

| 参数 | 行为 |
|------|------|
| `async=false`（默认） | HTTP 请求阻塞直到任务完成，直接返回 success/failed |
| `async=true` | 立即返回 `pending`，后台线程执行，通过 GET jobs 轮询状态 |

**并发控制：**
- 同一 `kb_id` 下，若有 `running` 状态的任务，新的 ingest 请求返回 `409 KB_JOB_RUNNING`
- Watchdog：启动时扫描运行超过 120 秒的 `running` 任务，自动标记为 `failed`（进程异常退出恢复）

## 4. 接口列表

### 4.1 触发入库任务

```
POST /api/kb/{kb_id}/ingest
```

请求体：
```json
{
  "force_reindex": false,
  "async": false
}
```

- `force_reindex=true`：即使已有索引也重建
- `async=true`：异步执行，返回 `pending`

成功（同步）：
```json
{ "request_id": "req_xxx", "data": { "job_id": "job_xxx", "status": "success" } }
```

成功（异步）：
```json
{ "request_id": "req_xxx", "data": { "job_id": "job_xxx", "status": "pending", "async": true } }
```

错误（已有运行中任务）：
```json
{ "error": "kb has running job", "code": "KB_JOB_RUNNING" }
```

---

### 4.2 查询任务列表

```
GET /api/kb/{kb_id}/jobs?limit=20&offset=0
```

返回按 `created_at DESC` 排序的任务列表，包含 `summary` 便捷字段：
- `success: indexed N chunks`
- `failed: <error excerpt>`
- `pending` / `running` / `cancelled`

---

### 4.3 查询单任务详情

```
GET /api/kb/{kb_id}/jobs/{job_id}
```

返回完整字段：`status`, `retry_count`, `last_error`, `payload`, `result`, `summary`, 时间戳。

---

### 4.4 取消任务

```
POST /api/kb/{kb_id}/jobs/{job_id}/cancel
```

- `pending` → `cancelled`（可取消）
- `running` → 仅标记 DB 为 `cancelled`，后台线程不强杀（当前限制）
- `success/failed/cancelled` → 返回当前状态，`cancelled: false`

---

### 4.5 重试任务

```
POST /api/kb/{kb_id}/jobs/{job_id}/retry
```

- 复用原任务参数（`payload_json`）
- `retry_count + 1`
- 重置 `status = pending`，同步执行
- 若 kb 下有 `running` 任务，返回 `409 KB_JOB_RUNNING`

---

### 4.6 手动运行任务

```
POST /api/kb/{kb_id}/jobs/{job_id}/run
```

- 适用于 `pending/cancelled/failed` 状态的任务
- 支持 `{ "async": true }` 参数

---

## 5. JobExecutor 设计

当前实现：`custom_app/services/job_executor.py`

- 单 Worker 线程（`max_workers=1`），FIFO 队列
- 线程以 daemon 模式运行，进程退出自动回收
- 执行异常由任务函数内部捕获并写 DB，不向外抛出

**Phase 2.5 升级方向：**
- 可配置并发数（`max_workers > 1`）
- 支持任务超时强杀（`threading.Timer` + `Thread.join(timeout)`）
- 任务队列持久化（重启后恢复 pending 任务）

---

## 6. 错误处理约定

| 场景 | HTTP 状态 | 错误码 |
|------|-----------|--------|
| KB 不存在 | 404 | `KB_NOT_FOUND` |
| 已有运行中任务 | 409 | `KB_JOB_RUNNING` |
| 任务不存在 | 404 | `JOB_NOT_FOUND` |
| 任务正在运行 | 409 | `JOB_RUNNING` |
| 执行失败（同步） | 500 | `INGEST_FAILED` |

---

## 7. 已知限制（Phase 2 v1）

1. **cancel 不能真正停止运行中的线程**：当前只修改 DB 状态，实际线程继续执行到完成
2. **无文件上传接口**：文件需手动放置到 `data/kb/<kb_id>/raw/`
3. **无任务超时强杀**：依赖 Watchdog（120s）事后标记，不是主动中止
4. **路径依赖 CWD**：`data_path` 使用相对路径，需从项目根目录启动

- [x] 取消 pending 任务，状态变 `cancelled`

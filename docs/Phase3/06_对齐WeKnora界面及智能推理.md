# Phase 3+ 大任务：对齐 WeKnora 界面（会话列表）与智能推理（含回复慢排查）

> **文档性质**：独立规划与实施路线图；**不包含**具体代码。  
> **依据**：[`../WeKnora智能推理与UltraRAG移植指南.md`](../WeKnora智能推理与UltraRAG移植指南.md)（下称《移植指南》）、[`04_API对接设计.md`](04_API对接设计.md)、[`03_页面功能设计.md`](03_页面功能设计.md)、当前 `custom_app` 实现现状。  
> **状态**：**规划已确认**（2026-04-27）——按本文阶段实施并拆 PR；§7 中**分项 gate**（会话落库方式、`agent` 一期选型、`04`/`03` 修订版）在对应阶段开工前补勾。  
> **说明**：**子任务 C（回复慢）** 与界面/智能推理无强依赖，建议 **P0 优先**、与 Phase 0 **并行**启动排查（与本次整体确认一致，可先行 **Phase P**）。

## 1. 需求重述（与现状差距）

### 1.1 子任务 A — 会话列表（WeKnora 式侧栏）

| 目标 | 说明 |
|------|------|
| **产品表现** | 左侧栏除「知识库 / 新建对话」外，提供**可点击的会话历史**（参考 WeKnora：按时间分组如「近 30 天」、每条显示标题摘要、点击进入该会话）。 |
| **当前现状** | `index.html` 仍为 **Sprint 1 占位文案**（「后续接入多会话」）；无会话 id、无列表数据源、无切换路由。 |
| **验收方向** | 选一会话 → 主区加载该会话消息；新建会话 → 新线程；与当前 `kb_id`、智能体模式等**组合策略**需在设计中定稿（见 §4）。 |

### 1.2 子任务 B — 智能推理（非仅 UI）

| 目标 | 说明 |
|------|------|
| **产品表现** | 用户选择「智能推理」时，执行路径与「快速问答」**可区分**：多轮检索/读全文/工具（或最小版编排），并通过 SSE 驱动前端的**推理步骤区**（与《移植指南》中 WeKnora 的 Think / Tool / Observe 可观测性对齐）。 |
| **当前现状** | 前端已传 `agent_mode`（`quick` / `agent`），但 **`custom_app/api/chat.py` 未读取该字段**，`RagRunner.chat_stream` **无分支**；与 WeKnora `smart-reasoning` 的 ReAct 管线**不等价**。 |
| **验收方向** | 至少满足 [`04_API对接设计.md`](04_API对接设计.md) 中的契约：**真 `agent` 或显式 `meta` 降级**（二选一在 §3 冻结）；长期对齐《移植指南》层 A → B → C。 |

### 1.3 子任务 C — 回复慢故障排查（快速问答）

| 目标 | 说明 |
|------|------|
| **现象** | **快速问答**下，用户提交问题后约 **10 分钟级** 才出现首字/首包或整段回复，体验不可接受（与「智能推理多轮」无关时也应先排除）。 |
| **目标** | 端到端量化延迟（T0 请求 → T1 首条 SSE → T2 首 chunk → T3 done），定位瓶颈在 **网络 / 网关 / Flask / 嵌入与检索 / 重排 / LLM 生成 / 流式缓冲** 哪一段；给出**可复现**结论与修复或配置项（超时、并发、模型路由、索引体量等）。 |
| **验收方向** | 在内网基准环境（指定 `kb_id`、典型问题）下，**首 token 或首条可见 SSE** 延迟达到与业务一致的 SLA（需在 Phase P 冻结具体数值，如 P95 &lt; 30s 等）；并保留排查记录（日志片段、配置前后对比）写入 §8。 |

## 2. 与《移植指南》的映射关系

| 《移植指南》分层 | 含义（摘要） | 与本大任务的关系 |
|------------------|-------------|------------------|
| **层 A — Deep Read 语义** | 首轮检索后强制拉全文 chunk 再生成，仍为**单轮** | 可作为 **`agent` 与 `quick` 的第一档差异**（实现成本相对低），或作为两者共用的质量增强 |
| **层 B — 轻量多轮编排** | 固定 K 轮、预定义工具、SSE 步骤事件 | **智能推理**的推荐中期目标；需扩展 `rag_runner` / SSE 类型 |
| **层 C — 完整 Agent** | 工具注册表、并行、压缩、Skills 等 | **独立立项**；本规划仅做依赖评估与接口预留，不默认一期交付 |

**推荐推进顺序**（与《移植指南》§7 一致，缩写到本任务）：

0. **（P0，可与下文并行）** **子任务 C**：按 **§3 Phase P** 做快速问答延迟排查，避免在「极慢」基线上叠新功能。  
1. 冻结 **会话模型 + API** 与 **`agent_mode` 行为**（文档先行：`04` 补丁、`03` 线框更新）。  
2. **智能推理**：优先落地 **层 A 或「meta 显式降级」** 二选一的可验收组合；再评估层 B 原型。  
3. **会话列表**：与 Phase 2 `app.sqlite` 扩展或新表设计并行，避免前端假数据长期存在。

### Phase P — 快速问答「回复极慢」端到端排查（**建议优先**，0.5～3 天，视根因）

> 与 Phase 0 **无强依赖**，可单独排期；根因在 LLM 侧时可能仅需改配置或换部署，不涉及会话/Agent 代码。

| 步骤 | 内容 | 产出 |
|------|------|------|
| P.1 | **复现与分层**：同一条问题固定 `kb_id`；浏览器 Network（TTFB、`/api/chat/stream`）、服务端日志时间戳；区分「长时间无 SSE」vs「有 status/chunk 但极稀疏」 | 现象归类（等待队列 / 同步阻塞 / 生成慢 / 代理超时 等） |
| P.2 | **服务端分段计时**：在 `chat_stream` / `RagRunner.chat_stream`（或 `_prepare_chat_context`、`_generate_stream`）增加**可开关**的耗时日志（检索、rerank、首 token）；避免生产默认刷屏 | 各段 P95 数据 |
| P.3 | **常见根因清单**（逐项排除）：嵌入 API 超时或限流；FAISS/磁盘 IO；单次 prompt 过大；`servers/generation` 指向的模型排队或 `max_tokens`/温度无关的排队；反向代理缓冲未关闭导致 SSE 延迟；DNS/跨网段调用外网模型 | 勾选表 + 证据 |
| P.4 | **缓解与回归**：配置调优（超时、`top_k`、关 rerank 试对比）、限流与扩容建议；必要时增加首包前 **心跳/进度** SSE（产品决策） | 变更说明 + 复测结果写入 §8 |
| P.5 | **SLA 定稿**：与业务约定「快速问答」可接受的上限（首字/完成），写入 `04` 或运维手册 | 供 §9 验收 |

## 5. 风险与缓解

| 风险 | 级别 | 缓解 |
|------|------|------|
| 会话与 KB 切换组合导致状态混乱（消息串库） | **高** | Phase 0 用状态机表写清；E2E 覆盖「换 kb + 换 session」 |
| `agent` 与 `quick` 行为无差异引发用户不信任 | **高** | 必须显式 `meta` 或 UI 提示；层 A 尽快做出可感知差异 |
| 层 B JSON 解析与工具安全（注入、越权读文件） | **高** | 白名单工具、参数校验、与 Phase 2 路径校验一致 |
| SQLite 写放大与并发 | **中** | 会话消息批量写、索引、`WAL` 模式评估 |
| 移动端侧栏空间不足 | **中** | 会话列表可折叠二级抽屉或与 WeKnora 折中布局 |
| **快速问答端到端极慢（10 分钟级）** | **高（体验）** | 优先 Phase P；易误判为「前端卡」实为服务端或模型排队；无 SLA 则难以验收后续功能 |

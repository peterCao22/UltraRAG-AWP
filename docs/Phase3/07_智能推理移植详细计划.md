## 一、两系统架构对比（现状速览）

### 1.1 核心差距一览

| 维度 | UltraRAG（现状） | WeKnora（目标参考） | 差距评级 |
|------|-----------------|-------------------|---------|
| **推理模式** | 单轮 RAG（检索→生成） | ReAct 多轮循环（Think→Act→Observe） | ★★★★★ |
| **工具系统** | 无（只有 FAISS 检索） | 15 种工具（搜索/分析/图谱/技能脚本等） | ★★★★★ |
| **SSE 事件** | 5 种（status/chunk/sources/meta/done） | 8 种（+thought/tool_call/tool_result） | ★★★☆☆ |
| **Agent 模式** | 层 A（全文 chunk 扩展） | 完整 ReAct 循环 | ★★★★☆ |
| **会话状态** | 落库但不参与推理 | 全对话历史参与 Agent 上下文 | ★★★☆☆ |
| **前端可视化** | 无推理步骤展示 | 实时 Think/Tool/Observe 步骤区 | ★★★★☆ |
| **Token 管理** | 无 | 双估算 + LLM 压缩 + 上下文窗口管理 | ★★★☆☆ |
| **并发工具执行** | 无 | 并行 Tool Call（asyncio） | ★★☆☆☆ |

### 1.2 WeKnora 智能推理核心机制（源码精要）

WeKnora 的智能推理是标准 **ReAct（Reason + Act）** 模式，核心在 `internal/agent/engine.go`：

```
每轮迭代：
┌─ THINK  → 调用 LLM（含 function calling），流式输出推理过程
│           事件：EventAgentThought（实时 chunk）
├─ ACT    → 并行执行 LLM 决定调用的工具
│           事件：EventAgentToolCall（调用前）、EventAgentToolResult（完成后）
├─ OBSERVE→ 工具结果追加到 messages（OpenAI format）
│           无直接 SSE；结果进入下一轮 THINK 的上下文
└─ 判断   → finish_reason=stop 且无 tool_calls → 结束；或显式调用 final_answer 工具
```

**工具清单**（`internal/agent/tools/definitions.go`）：

| 工具内部名 | 前端显示 | 功能 |
|-----------|---------|------|
| `thinking` | 深度思考 | 动态反思性思考 |
| `todo_write` | 制定计划 | 结构化研究计划 |
| `grep_chunks` | 关键词搜索 | 精确词匹配 chunk |
| `knowledge_search` | 语义搜索 | 向量检索 |
| `list_knowledge_chunks` | 查看文档分块 | 获取完整 chunk |
| `query_knowledge_graph` | 查询知识图谱 | 知识图关系 |
| `get_document_info` | 获取文档信息 | 文档元数据 |
| `final_answer` | 最终回答 | 提交最终答案 |

```python
    """发射 thought 事件（用于前端推理步骤区）"""

def _emit_tool_result(self, tool_name: str, result_summary: str, duration_ms: int) -> dict:
```

> **目标**：实现真正的多轮推理循环，支持有限工具调用，对标 WeKnora 层 B。

### B.1 架构设计

```
custom_app/
├── services/
│   ├── rag_runner.py          （现有，单轮 RAG，quick 模式继续使用）
```

```python
async def knowledge_search(query: str, top_k: int = 5, runner: RagRunner = None) -> str:
    """
    对知识库执行语义向量搜索。
    复用 RagRunner._embed_query + _index.search + _rerank_hit_ids

    hit_ids = runner.rerank(query, hit_ids)[:top_k]
    chunks = [runner.get_chunk(i) for i in hit_ids]
    return format_chunks_for_llm(chunks)
```

### B.3 ReAct 循环引擎（`services/agent_runner.py`）

参考 WeKnora `internal/agent/engine.go` 的 `executeLoop`，Python 异步版：

```python
class AgentRunner:
    MAX_ITERATIONS = 10  # 一期保守值；WeKnora 默认 20

    def __init__(self, runner: RagRunner, llm_client, registry: ToolRegistry):
        self.runner = runner
        self.llm = llm_client
        self.registry = registry

    async def run(self, question: str, history: list[dict]) -> AsyncGenerator[dict, None]:
        """
        主入口：执行 ReAct 循环，yield SSE 事件字典。
        """
        messages = self._build_initial_messages(question, history)
        tools = self.registry.get_tool_schemas()

        for iteration in range(self.MAX_ITERATIONS):
            # ── THINK ──
            yield {"type": "thought", "content": ""}  # 开始新一轮思考
            thought_chunks = []
            tool_calls = []

            async for chunk in self._call_llm_stream(messages, tools):
                if chunk["type"] == "thought_chunk":
                    thought_chunks.append(chunk["content"])
                    yield {"type": "thought", "content": chunk["content"]}
                elif chunk["type"] == "tool_call":
                    tool_calls.append(chunk)

            full_thought = "".join(thought_chunks)

            # ── 停止判断 ──
            if not tool_calls:
                # LLM 无工具调用 → 直接生成最终答案
                yield {"type": "final_answer", "content": full_thought}
                return

            # ── ACT ──
            tool_results = []
            for tc in tool_calls:
                tool_name = tc["name"]
                tool_args = tc["args"]
                hint = self._format_tool_hint(tool_name, tool_args)

                yield {"type": "tool_call", "tool_name": tool_name,
                       "display_name": self.registry.get_display_name(tool_name),
                       "hint": hint}

                # final_answer 特殊处理
                if tool_name == "final_answer":
                    yield {"type": "final_answer", "content": tool_args.get("answer", "")}
                    return

                t0 = time.monotonic()
                result, _ = await self.registry.execute(tool_name, tool_args)
                duration_ms = int((time.monotonic() - t0) * 1000)

                yield {"type": "tool_result", "tool_name": tool_name,
                       "display_name": self.registry.get_display_name(tool_name),
                       "summary": result[:200],  # 截断摘要
                       "duration_ms": duration_ms}

                tool_results.append({"tool_call_id": tc["id"], "result": result})

            # ── OBSERVE ──
            messages = self._append_tool_results(messages, tool_calls, tool_results)

        # 超出最大轮次 → 合成答案
        yield {"type": "thought", "content": "已达到最大推理轮次，综合已有信息生成答案..."}
        final = await self._synthesize_answer(messages, question)
        yield {"type": "final_answer", "content": final, "fallback": True}
```

**关键实现细节**：

1. **Messages 格式**（兼容 OpenAI function calling）：
```python
def _build_initial_messages(self, question: str, history: list) -> list[dict]:
    system = self._build_system_prompt()
    msgs = [{"role": "system", "content": system}]
    # 注入会话历史（最近 N 轮）
    for turn in history[-6:]:  # 保守值：6 轮
        msgs.append({"role": "user", "content": turn["user"]})
        msgs.append({"role": "assistant", "content": turn["assistant"]})
    msgs.append({"role": "user", "content": question})
    return msgs

def _append_tool_results(self, messages, tool_calls, results) -> list:
    # OpenAI 格式：assistant message（含 tool_calls）+ tool messages
    assistant_msg = {
        "role": "assistant",
        "content": None,
        "tool_calls": [{"id": tc["id"], "type": "function",
                        "function": {"name": tc["name"],
                                     "arguments": json.dumps(tc["args"])}}
                       for tc in tool_calls]
    }
    messages = messages + [assistant_msg]
    for tc, r in zip(tool_calls, results):
        messages.append({
            "role": "tool",
            "tool_call_id": tc["id"],
            "content": r["result"]
        })
    return messages
```

2. **System Prompt 设计**（对齐 WeKnora 完整工作流）：

WeKnora 的 System Prompt 是 Agent 效果的核心，采用 **"Assess-Reconnaissance-Plan-Execute"（评估-侦察-计划-执行）** 四阶段工作流。UltraRAG 的 System Prompt 需完整对应此结构，并针对 AGV 工业场景定制。

**System Prompt 模板变量**（运行时注入）：
```
{{kb_name}}         — 当前知识库名称
{{current_time}}    — 当前时间（ISO 格式）
{{language}}        — 用户语言（zh-CN / en 等，自动检测）
{{web_search_status}} — Web 搜索是否启用（UltraRAG 一期固定为"未启用"）
```

**System Prompt 完整结构**（UltraRAG 适配版）：

```
### Role
你是一个专业的工业设备知识库智能助手，基于渐进式 Agentic RAG。
核心理念是"证据优先"：不依赖内部参数知识，仅基于从知识库检索到的经过验证的内容构建答案。

### Critical Constraints（必须遵守）
1. 每次 grep_chunks 或 knowledge_search 返回结果后，必须立即调用 list_knowledge_chunks
   读取对应 chunk 的完整内容，不得仅凭搜索摘要作答。
2. 在所有用户可见输出（包括思考过程）中，使用自然语言描述替代工具内部名称：
   - 说"搜索知识库"而非"knowledge_search"
   - 说"阅读文档内容"而非"list_knowledge_chunks"
   - 说"文本搜索"而非"grep_chunks"
3. 不得在思考过程或答案中暴露内部 ID（chunk_id、knowledge_id 等），
   用文档标题或名称代替。
4. 必须以调用 final_answer 工具结束，不得直接输出最终答案。

### Workflow
#### Phase 0：意图评估（任何检索前）
判断是否需要检索：
- 纯对话（问候/致谢）→ 直接 final_answer
- 需要文档知识 → 进入 Phase 1

#### Phase 1：初步侦察（Deep Read）
1. 执行 grep_chunks（关键词）和 knowledge_search（语义）
2. 对返回的 ID，立即调用 list_knowledge_chunks 读取完整内容
3. 在思考中评估：内容是否充分回答问题？

#### Phase 2：策略决策
- 路径 A（直接回答）：完整内容已足够 → 进入 Phase 4
- 路径 B（复杂研究）：需多步检索 → 调用 todo_write 制定计划 → 进入 Phase 3

#### Phase 3：按计划执行（循环）
对 todo_write 中每个任务：
1. 执行检索（grep_chunks / knowledge_search）
2. 必须调用 list_knowledge_chunks 读取完整内容
3. 在思考中反思：内容是否满足子任务？有无缺口？
4. 补充检索或标记完成

#### Phase 4：综合生成
所有任务完成后，综合全部 chunk 完整内容，调用 final_answer 提交答案。

### 检索策略（严格顺序）
1. grep_chunks（关键词锚定，1-3 个关键词）
2. knowledge_search（语义扩展）
3. list_knowledge_chunks（必须执行，读取完整内容）
4. query_knowledge_graph（可选，关系探索）

### 最终输出规范
- 所有事实性陈述须内联引用：<kb doc="文档名" chunk_id="xxx" />
- 引用标签紧跟所在段落末尾，不换行，不集中放在结尾
- 如检索到图片，使用 Markdown 图片语法嵌入：![描述](图片URL)

### 系统状态
当前时间：{{current_time}}
Web 搜索：{{web_search_status}}
用户语言：{{language}}
知识库：{{kb_name}}
```

**实现要点**（`agent_runner.py` 的 `_build_system_prompt` 方法）：
- 模板存放在 `prompt/agv_agent_system.jinja`（复用现有 Jinja2 基础设施）
- `language` 从请求 header 或 `Accept-Language` 自动提取，默认 `zh-CN`
- `web_search_status` 一期固定为 `"未启用"`

3. **LLM 流式调用**：需要解析 stream 中的 `tool_calls`（Gemini function calling 的响应格式，与 OpenAI delta 有差异，需专门适配器，见 §5.2）。

## 五、关键技术问题与解决方案

### 5.1 Flask 同步 vs 异步（重要）

**问题**：AgentRunner 的多轮 LLM 调用天然是异步的，但现有 Flask 是同步 WSGI。

**方案选项**：

| 方案 | 改动量 | 推荐度 |
|------|-------|-------|
| A：在 Flask 路由中用 `asyncio.run()` 运行每轮 | 低（无架构变动） | ★★★★☆ |
| B：迁移到 Flask + `asgiref`（ASGI 兼容层） | 中 | ★★★☆☆ |
| C：新起一个 FastAPI 服务，Flask 反代 | 高 | ★★☆☆☆（overkill） |

**建议**：阶段 B 采用方案 A（`asyncio.run` 包装），架构不动，等阶段 C 再评估是否升级。

**实现**：
```python
import asyncio

def _stream_agent_response(payload):
    def generate():
        async def _run():
            async for event in agent_runner.run(...):
                yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
        # 在新 event loop 中运行
        loop = asyncio.new_event_loop()
        try:
            gen = _run()
            while True:
                chunk = loop.run_until_complete(gen.__anext__())
                yield chunk
        except StopAsyncIteration:
            pass
        finally:
            loop.close()
    return Response(generate(), mimetype="text/event-stream")
```

### 5.2 Gemini Function Calling 适配器

**已确认**：当前测试环境使用 Gemini，两种模型均支持 function calling。但 Gemini API 的响应格式与 OpenAI 有以下关键差异，`agent_runner.py` 需要专门适配：

| 项目 | OpenAI 格式 | Gemini 格式 |
|------|------------|------------|
| 工具定义传参 | `tools: [{type: "function", function: {...}}]` | `tools: [{function_declarations: [...]}]` |
| 工具调用响应 | `message.tool_calls[].function.name/arguments` | `candidates[0].content.parts[].function_call.name/args` |
| 工具结果传回 | `role: "tool"` + `tool_call_id` | `role: "function"` + `name` |
| 流式 delta 拼接 | `delta.tool_calls[].function.arguments`（增量拼接） | `delta.function_call.args`（增量，同样需拼接） |

**适配器设计**（`services/llm_adapter.py`，新建）：

```python
class GeminiAdapter:
    """将 Gemini API 响应统一转换为内部 tool_call 格式"""

    def build_tools_param(self, registry: ToolRegistry) -> list:
        """转换为 Gemini function_declarations 格式"""
        return [{"function_declarations": [
            {"name": t.name, "description": t.description,
             "parameters": t.parameters}
            for t in registry.all()
        ]}]

    def parse_stream_chunk(self, chunk) -> dict:
        """解析单个流式 chunk，返回统一内部格式"""
        # 返回：{"type": "thought_chunk"|"tool_call_delta"|"finish", ...}
        ...

    def build_tool_result_message(self, tool_call_id: str,
                                  name: str, result: str) -> dict:
        """构造 Gemini 格式的工具结果 message"""
        return {"role": "function", "name": name,
                "content": result}
```

**vLLM Qwen3.6 备用**：vLLM 暴露 OpenAI 兼容接口，格式与 OpenAI 一致，无需额外适配。切换只需改 `ULTRARAG_CHAT_BACKEND=openai` 环境变量。

### 5.5 MCP 工具扩展支持

**问题**：WeKnora 的工具配置页支持 MCP（Model Context Protocol）工具；UltraRAG 本身的核心架构就是基于 MCP 的（`servers/retriever`、`servers/generation` 等均为 MCP Server）。

**UltraRAG 与 MCP 的关系**：

```
UltraRAG 已有 MCP 基础设施：
├── servers/retriever    → MCP Server（检索）
├── servers/generation   → MCP Server（生成）
├── servers/corpus       → MCP Server（解析）
└── MCP Client（ultrarag core）→ 编排以上 servers
```

**Agent 工具与 MCP 的整合路径**：

- **阶段 B（一期）**：Agent 工具（`knowledge_search` 等）**直接调用** UltraRAG MCP Client 的内部 Python API，不走 HTTP/MCP 协议（性能优先，无额外网络开销）
- **阶段 C（扩展）**：将外部 MCP Server 注册为 Agent 工具，通过标准 MCP 协议调用第三方能力（如数据库查询、外部 API、自定义业务系统）

**阶段 C 预留接口**（`services/tools/registry.py` 中预留）：

```python
class ToolRegistry:
    def register_mcp_server(self, server_url: str, tool_names: list[str]):
        """将外部 MCP Server 的工具动态注册到 registry（阶段 C 实现）"""
        raise NotImplementedError("Phase C")
```

**结论**：一期不需要改动 MCP 层；阶段 C 通过 `register_mcp_server` 扩展，届时 UltraRAG 的 MCP 架构反而是优势——其他 MCP Server 可直接复用。

### 5.6 用户可见输出规范（禁止暴露内部细节）

**来源**：WeKnora System Prompt 中的 Critical Constraint 5 — 所有用户可见输出必须用自然语言，不得暴露工具内部名称和 ID。

**UltraRAG 需要遵守的规则**：

| 禁止 | 应该说 |
|------|-------|
| `knowledge_search("AGV 故障")` | `搜索知识库："AGV 故障"` |
| `list_knowledge_chunks(chunk_id=42)` | `阅读文档《AGV操作手册》` |
| `grep_chunks` | `文本搜索` |
| `chunk_id: c_0042` | `《AGV操作手册》第3章` |

**影响两个层面**：

1. **后端 SSE 的 `tool_call.hint` 字段**：`agent_runner.py` 的 `_format_tool_hint` 方法需将工具调用翻译为用户友好文案（已在 B.3 节设计中包含 `display_name` 字段，此处是其具体落地规则）。

```python
def _format_tool_hint(self, tool_name: str, args: dict) -> str:
    """生成用户友好的工具调用描述，不暴露内部名称"""
    hints = {
        "knowledge_search": lambda a: f'搜索知识库："{a.get("query", "")[:30]}"',
        "keyword_search":   lambda a: f'文本搜索："{a.get("keywords", "")[:30]}"',
        "list_knowledge_chunks": lambda a: f'阅读文档：{self._doc_title(a.get("doc_id",""))}',
        "final_answer":    lambda a: "提交最终答案",
    }
    return hints.get(tool_name, lambda a: tool_name)(args)
```

2. **前端 `thought` 事件渲染**：LLM 的思考内容（`thought` SSE）直接展示给用户，需依赖 System Prompt 约束 LLM 自我审查；前端不做额外过滤（过滤会破坏流式体验）。

### 5.7 内联引用格式（`<kb doc="..." />`）

**WeKnora 的做法**：LLM 答案中使用 `<kb doc="文档名" chunk_id="xxx" />` 内联引用标签，前端解析并渲染为可点击的来源链接。

**UltraRAG 当前做法**：答案生成后统一在末尾返回 `sources` 数组（SSE `sources` 事件），由 `sourcePanel.js` 渲染来源面板，引用非内联。

**是否对齐 WeKnora 的内联引用**：

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| 保持现有 `sources` 面板 | 无需改 LLM prompt 和前端解析器 | 来源与答案文本分离，不如内联直观 | **阶段 B 保留** |
| 对齐 WeKnora 内联 `<kb />` | 来源与上下文紧密关联，专业感强 | 需要前端 HTML 解析器 + 安全过滤 | **阶段 C 可选** |

**阶段 B 的折中方案**：在 System Prompt 中要求 LLM 在答案段落末尾标注 `【来源：文档标题】`（纯文本，无 HTML），前端通过正则识别并高亮显示，同时保留 `sources` 面板作为完整来源列表。

### 5.8 工具输出的安全性

**WeKnora 的做法**：白名单工具 + 参数校验 + 输出截断 + 错误提示注入。

**UltraRAG 对应措施**：
- 所有工具只能访问当前 `kb_id` 对应的数据，不得越权读取其他 KB
- 工具输出最长 3000 字符（防止 prompt injection 攻击）
- 工具执行超时：单次最多 30 秒
- 工具错误：返回结构化错误信息给 LLM（不抛异常终止循环）

### 5.5 回复慢问题（Phase P 关联）

**Agent 模式会比 quick 模式更慢**（多轮 LLM 调用），因此必须先解决 Phase P 的基础延迟。

**Agent 模式预期延迟估算**：

| 步骤 | 预期耗时 |
|------|---------|
| Embedding（query） | 0.5～2s |
| FAISS 检索 | <0.1s |
| CrossEncoder Rerank | 1～5s |
| LLM 单轮（思考+工具调用） | 5～15s（首 token） |
| 工具执行 | 0.1～2s |
| 总计（2 轮） | 15～40s |

**建议**：
- `agent` 模式需要更长的客户端超时（当前 300s 够用）
- 第一个 `thought` SSE 应在 5s 内发出（体验关键）
- 必要时加入"心跳"SSE（每 10s 发一次 `status: "推理中..."`）

## 七、阶段 C 接口预留（不在一期实现）

以下接口在阶段 A/B 设计时需预留扩展点，避免阶段 C 大范围重构：

| 预留点 | 阶段 A/B 做法 | 阶段 C 扩展 |
|-------|-------------|-----------|
| 工具注册 | `ToolRegistry.register()` 单点注册；`enabled_tools` 列表过滤 | `register_mcp_server()` 动态注册外部 MCP Server 工具 |
| SSE 事件类型 | `thought/tool_call/tool_result/final_answer` | 加入 `reflection`、`plan`（todo_write 步骤事件） |
| 并发工具执行 | 串行（简单） | `asyncio.gather()` 并行执行多个工具 |
| Token 管理 | 硬截断（2000 字符）+ 轮次限制（6 轮） | LLM 摘要压缩 + BPE 本地估算（对齐 WeKnora） |
| 记忆系统 | 无 | Session 级长期记忆（跨对话上下文压缩） |
| 多知识库 | 单 `kb_id` | `AgentConfig.kb_ids` 数组，跨库联合检索 |
| 内联引用格式 | `【来源：文档名】` 纯文本标注 | `<kb doc="..." chunk_id="..." />` HTML 标签 + 前端解析器 |
| 工具页面配置 | API 参数 `enabled_tools` 控制 | 管理页面（`admin.html`）可视化勾选 + `agent_configs` 表持久化 |
| Skills 技能脚本 | 不实现（RAG 场景收益有限） | 阶段 C 单独评估，参考 WeKnora `skills.Manager` |

## 八、风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 现有 LLM 不支持 Function Calling | 高 | 阻断阶段 B | 优先排查；备选方案：XML 格式工具调用 |
| Agent 模式首 token 延迟 >30s | 中 | 体验差 | Phase P 先行；心跳 SSE；思考流式输出 |
| Flask 同步环境阻塞 SSE | 中 | 推理步骤延迟显示 | 线程隔离；评估 ASGI |
| Tool 调用死循环（LLM 重复调同一工具） | 中 | 超 MAX_ITER | 工具调用去重检测 + 强制终止 |
| Prompt Injection（工具结果注入） | 低 | 安全 | 输出截断 + 内容过滤 |
| 阶段 A 模拟推理被识别为"假" | 低 | 信任损失 | UI 标注"层 A 增强模式"；阶段 B 尽快跟进 |

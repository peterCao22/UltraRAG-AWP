# Phase 12 —— 对话智能化（Conversation Intelligence）

> **状态**：方向锚点（2026-05-16），详细计划待 Phase 11 完成后展开
> **前置**：[Phase 11](../Phase11/README.md) 全部完成
> **定位**：让对话**像人**对话 —— 多轮上下文 / 指代消解 / 长会话记忆 / 主动澄清
> **工时估算**：4-6 周

---

## 一、为什么单独成 Phase

Phase 8-11 累积下来的能力：

- ✅ 单轮检索准（Phase 8）
- ✅ 图文联动（Phase 9）
- ✅ 多租户共享（Phase 10）
- ✅ 生产化 + 体验扩展（Phase 11）

**但对话本身还是「一问一答」式**，做不到：

- 用户问"它怎么处理"，系统不知道"它"指什么
- 长对话后，前面说过的关键信息被遗忘
- 用户问得模糊时，系统不会反问澄清
- Agent 多轮调工具时，上下文跨工具丢失

这一组能力**主题统一**（多轮对话智能），**前置依赖一致**（Phase 11.1 审计日志 + 11.2 Follow-up），**评测维度一致**（多轮对话准确率），应独立成档。

## 二、子阶段拆分

### Phase 12.1 — Context Resolution（指代消解）（1.5-2 周）

**目标**：解决"它 / 这个 / 上一步 / 第 3 项"等代词指代

**核心思路**：

```
用户：AGV 启动前要做什么？
系统：1. 检查电池 2. 检查急停按钮 3. ...

用户：第 2 个怎么操作？     ← 这里"第 2 个"指什么？
系统应理解为：             ← Context Resolution
  - 第 2 个 = 检查急停按钮
  - 改写 query 为"急停按钮如何检查"
  - 走检索 + 生成
```

**实现路径**：
- 每轮回答完成时，LLM 抽取"可指代锚点"（实体 / 列表项 / 步骤号）存进 session 上下文
- 新 query 来时，先用 LLM **判断是否含指代**；如有，用历史锚点改写
- 改写后的 query 走正常 RAG 链路

**关键风险**：改写错了比不改写更糟。需要 confidence threshold + fallback 到原 query。

---

### Phase 12.2 — Session Memory（长会话摘要）（1 周）

**目标**：用户聊了 30 轮后，前面的关键信息不丢失

**问题现状**：
- 现在 chat 每次把所有历史塞进 prompt，长会话 token 爆掉
- 简单截断（只取最近 N 轮）会丢失关键前置信息

**实现路径**：
- 每 N 轮（如 10 轮）触发摘要：用 LLM 把前段对话压缩成 200 字摘要
- 摘要存进 `kb_sessions.summary` 字段
- 后续 prompt 拼接：`[摘要] + [最近 N 轮] + [当前 query]`
- 参考 WeKnora [`llmcontext/context_manager.go`](../../../WeKnora/internal/application/service/llmcontext/context_manager.go) + WeKnora `memory/service.go`

**关键风险**：摘要丢失细节。评测：摘要前后回答的一致性。

---

### Phase 12.3 — Clarification（系统主动反问）（1 周）

**目标**：query 模糊时，系统不瞎答，反问澄清

**触发条件**：
- 检索 top-k 命中分数都低（< 阈值）
- 命中多个明显不同主题的 chunk（说明 query 跨域）
- 意图分类（Phase 11.1.5）置信度低

**示例**：

```
用户：怎么充电？
系统：你是问 AGV 充电还是 IFS 系统中的充电流程？    ← 触发反问
用户：AGV
系统：[继续走 RAG]
```

**实现路径**：
- 检索后加判断层：是否需要反问
- 反问 prompt：列出可能含义，让用户选
- 用户回答后，把澄清信息加进 query 再次检索

**关键风险**：反问太频繁 = 烦人。需要严格触发条件 + Admin 可调阈值。

---

### Phase 12.4 — Multi-turn Agent 状态优化（1-2 周）

**目标**：Agent 模式下跨工具调用的上下文保持

**问题现状**：
- Agent 调 `knowledge_search` 找到 chunk A，再调 `query_kg` 找到实体 B
- 第三轮 Agent 决策时，A 和 B 之间的关系丢失（除非 LLM 自己记住）

**实现路径**：
- 引入 Agent **工作记忆**结构：每次工具调用后把结果摘要存进 session-level scratchpad
- LLM 决策时 prompt 加 scratchpad 段
- 工具调用历史可视化（用户能看到推理链）

**参考**：WeKnora [`agent/memory/consolidator.go`](../../../WeKnora/internal/agent/memory/consolidator.go) 实现

**关键风险**：scratchpad 膨胀。需定期 consolidate（合并相似条目）。

## 四、评测与退出条件

| 子阶段 | 关键评测 | 退出条件 |
|--------|---------|---------|
| 12.1 | 多轮对话 query 改写准确率（人工标注 50 条多轮场景） | ≥80% 改写正确 |
| 12.2 | 摘要前后回答一致性 | F1 不下降 |
| 12.3 | 反问触发的合理性 + 用户接受率 | 误触发 <10% |
| 12.4 | Agent 多轮任务完成率 | 比 Phase 11 基线提升 ≥10pp |

**最坏情况**：12 至 12.1 + 12.2 收尾（这两项收益最大）；12.3 / 12.4 可推迟。

## 五、关键风险

| 等级 | 风险 | 缓解 |
|------|------|------|
| 🔴 HIGH | 12.1 指代消解错了 = 答非所问 | confidence threshold；不确定时 fallback 到原 query；UI 上展示"系统理解你问的是 X，请确认" |
| 🟡 MED | 12.2 摘要丢失细节 | 关键信息（数字 / 实体）单独存索引；摘要文本作为补充 |
| 🟡 MED | 12.3 反问过频被嫌烦 | 默认保守阈值；Admin 可调；统计反问采纳率 |
| 🟡 MED | 12.4 scratchpad token 爆 | 定期 consolidate；超长强制摘要 |

## 六、给未来自己的备忘

写详细计划时**必须先确认**：

1. **多轮评测数据**：50 条多轮场景从哪来？业务侧手工写 + 从 Phase 11.1.2 审计日志抽
2. **改写后是否给用户看**：Phase 12.1 改写后的 query 是显示给用户（"系统理解为：X，是否正确？"）还是静默？建议**显示**（教育用户 + 给修正机会）
3. **session memory 何时触发摘要**：固定每 N 轮 vs 按 token 量 vs 按时间间隔
4. **clarification 阈值**：硬编码 vs 配置化 vs 每 KB 独立
5. **Agent scratchpad 是否对用户可见**：调试用 vs 隐藏（关系到 UI 复杂度）
